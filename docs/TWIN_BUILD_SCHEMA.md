# TwinBuild v1 정규형

## 무엇을 해결하는가

어댑터마다 캔버스 노드와 수정안을 직접 만들면 새 웹사이트, 물류 시스템, CRM을 붙일 때마다 검토 엔진과 UI까지 고쳐야 한다. `TwinBuild`는 공급자별 발견 결과와 캔버스 사이에 놓이는 공통 중간 형식이다.

처리 순서는 다음과 같다.

1. 어댑터가 허용된 원본에서 근거를 발견한다.
2. 어댑터가 발견 결과를 `TwinBuild`로 정규화한다.
3. 공통 대조 엔진이 현재 캔버스와 `TwinBuild`를 비교한다.
4. 엔진이 추가·교체 가능한 변경만 지문이 붙은 수정안으로 만든다.
5. 사용자가 미리 보고 승인한 수정안만 기존 적용 엔진으로 전달한다.

`TwinBuild`는 소스 코드나 사용자 DB 본문을 복제하는 저장소가 아니다. 파일 경로, 자원 이름, 근거 지문, 제한된 설명처럼 시스템 구조를 확인하는 데 필요한 참조만 담는다.

## v1 기록 종류

### `source`

어떤 어댑터가 어느 스냅샷을 정규화했는지 기록한다. 어댑터 계약 버전, 어댑터 버전, 엔진 스키마 버전이 모두 포함된다.

### `entities`

API, 서비스, 데이터베이스, 사람, 배포 환경 같은 실체다. 안정된 ID, 역할, 환경, 공급자, 외부 참조, 부모 실체, 신뢰영역, 초기 배치 힌트를 가진다.

### `parts`

실체가 제공하거나 사용하는 코드, 입력, 출력, 보기, 연결, 트리거, 키 참조다. 반드시 한 실체에 속하며 필요하면 조작 계약을 참조한다.

### `relations`

두 실체 또는 두 파츠 사이의 의미 있는 관계다. `reads`, `writes`, `calls`, `syncs_with` 같은 온톨로지 종류, 근거, 신뢰 게이트웨이, 캔버스 연결점 힌트를 가진다.

### `trustZones`와 `gateways`

로컬 기기, 사설망, 공개 클라우드, 외부 SaaS 같은 통제 경계를 표현한다. 서로 다른 신뢰영역의 실체를 연결할 때는 실제 통과 지점인 게이트웨이가 반드시 있어야 한다. 게이트웨이의 방향, 노출 범위, 프로토콜, 데이터 종류, 인증·권한·암호화도 기록한다.

### `evidence`

파일, 설정, 커넥터 결과, 실행 기록, 문서의 참조와 지문이다. 실제 비밀 키나 토큰 값은 허용하지 않는다. 모든 실체·파츠·관계·게이트웨이·조작은 존재하는 근거 ID만 참조할 수 있다.

### `operations`

읽기·쓰기·실행 능력의 선언이다. 대상, 승인 수준, 복구 가능 여부, 근거를 가진다. 선언만으로 실행 권한이 생기지는 않으며 실제 실행은 별도의 계획·승인·검증 계약을 통과해야 한다.

## 참조 무결성과 보안 규칙

- 모든 기록 ID는 한 빌드 안에서 고유하고 재검사 후에도 안정적이어야 한다.
- 부모, 파츠 소유자, 관계 양 끝, 근거, 조작 대상, 게이트웨이 참조가 실제로 존재해야 한다.
- 관계의 파츠 연결점은 해당 엔티티에 실제로 속해야 한다.
- 서로 다른 신뢰영역을 잇는 관계는 양쪽 영역과 일치하는 게이트웨이 없이는 빌드 자체가 거부된다.
- 근거와 파츠 참조에는 실제 키·토큰·개인키 리터럴을 넣을 수 없다.
- 같은 입력은 정렬 순서와 무관하게 같은 정규형 지문을 만든다.
- 지원하지 않는 미래 스키마는 추측해 읽지 않고 명시적으로 거부한다.

## 사용자 데이터 소유권

공통 대조 엔진은 다음 시스템 계약만 비교한다.

- 실체의 종류와 의미 필드
- 정규화된 파츠 계약
- 관계의 종류·근거·파츠 연결점
- 신뢰영역과 게이트웨이 참조

다음 항목은 사용자가 소유하므로 자동으로 덮어쓰거나 삭제하지 않는다.

- 노드 위치와 크기
- 사용자가 작성한 메모와 알 수 없는 추가 필드
- 사용자가 직접 추가한 파츠·노드·연결선
- 기존 검토 결정

실체 설명이 정규형과 다를 때도 현재 단계에서는 차이만 표시한다. 의미를 자동 교체하는 범용 작업은 아직 허용하지 않는다.

## 단계적 대조

부모 실체가 없는 자식 실체와 양 끝 실체가 없는 관계는 즉시 추가하지 않는다.

1. 먼저 부모 또는 양 끝 실체를 추가한다.
2. 다시 검사한다.
3. 파츠를 추가하거나 지문이 일치하는 기존 파츠를 교체한다.
4. 마지막으로 관계와 신뢰 게이트웨이를 추가한다.

이 순서로 한 수정안이 보이지 않는 선행 조건을 몰래 만들지 않으며, 각 단계가 현재 캔버스 상태를 다시 확인한다.

## 현재 참조 구현

- Workflow Canvas 시스템 지도: 28개 실체, 9개 파츠, 38개 관계, 1개 조작 계약을 `TwinBuild`로 정규화한다.
- 주문 서비스 골든 픽스처: 공개 클라우드 API와 사설 데이터 영역, DB 게이트웨이, 파츠 연결 관계를 같은 엔진으로 대조한다.
- 주문 서비스는 DB 실체와 API 파츠를 먼저 적용한 뒤 관계를 적용하면 수정안 0개 상태가 된다.
- 이 과정에서 사용자가 옮긴 API 노드 위치·크기·메모와 직접 추가한 파츠가 보존된다.

## 마이그레이션

현재 스키마 버전은 `1`이다. 시험 단계의 v0 이름은 다음과 같이 v1로 전진 변환한다.

- `nodes` → `entities`
- `capabilities` → `parts`
- `connections` → `relations`
- `zones` → `trustZones`
- `actions` → `operations`

지원하지 않는 버전은 자동 추정하지 않는다. 이후 버전에서는 버전별 순차 마이그레이션과 지원 기간을 추가해야 한다.

## 아직 남은 범위

- 정책, 이벤트, 위협, 실행 관측을 독립된 정규 기록으로 확장
- 엔티티 의미 필드의 안전한 부분 교체 계약
- 실제 두 번째 언어·프레임워크 저장소 어댑터
- 대규모 빌드의 증분 대조와 성능 한도
- 범용 조작의 계획·실행·검증·복구 계약
