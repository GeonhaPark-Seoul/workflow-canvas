# 코드 품질 점검 플레이북 (Audit Playbook)

> 사용자가 "점검하자 / 점검일 / audit"라고 말하면 이 절차를 그대로 실행한다.
> 목적: 빠르게 쌓인 코드가 쓰레기가 되기 전에, 원할 때 날 잡고 안전하게
> 대체·정리할 수 있는 상시 준비 상태를 유지한다.

## 1. 상시 규칙 (매 배치마다 — 점검일이 아니어도)

구현·검수 중 다음을 발견하면 **그 자리에서** `docs/governance/TECHNICAL_DEBT.md`의
`Code quality` 섹션에 `QUAL-NNN` 항목으로 기록한다 (지우지 않고 상태만 갱신 —
장부 규칙 동일):

- 지름길: 시간 때문에 택한 차선의 구현 (무엇이 최선이었는지 함께)
- 중복: 같은 일을 하는 코드가 2곳 이상
- 비대: 한 파일/함수가 책임을 너무 많이 가짐
- 사각지대: 테스트가 못 보는 동작 (이번 층 활성화 감지 버그처럼 "조건 자체가 테스트 안 됨")
- 대체 후보: 직접 구현했지만 검증된 표준/라이브러리로 바꾸는 게 나은 것 (도입은 별도 승인)

기록 형식은 기존 장부와 동일: Severity / Gate(release-blocker 여부) / Status /
Context / Exit criteria. 품질 항목의 Gate는 대부분 `none`(출시 비차단)이다.

## 2. 점검일 절차 (사용자 요청 시)

한 세션에서 순서대로 실행하고 결과를 하나의 점검 보고로 제출한다.

### A. 기계 검사 (전부 실행, 결과 수치 기록)
1. `npm test` — 실패·건너뜀 0 확인
2. `npm run build` — 번들 크기를 직전 점검 기록과 비교 (±10% 이상이면 원인 규명)
3. `npm run governance:check` + `npm run privacy:check`
4. 미사용 후보 스캔 (새 의존성 없이):
   - export되지만 import되지 않는 심볼: `grep` 기반 교차 확인
   - `src/`·`shared/`·`mcp/`에서 어디서도 참조되지 않는 파일
5. 크기 상위 파일 10개 (`wc -l` 기준) — 직전 점검 대비 증감

### B. 장부 리뷰
6. `TECHNICAL_DEBT.md`의 QUAL 전체 + `in-progress` 항목을 훑고
   상태가 현실과 맞는지 갱신
7. 이번 점검에서 새로 발견한 항목 추가

### C. 판정과 제안
8. 상위 리팩터 후보 3개 선정 — 각각 근거(중복 줄 수·크기·버그 이력),
   예상 효과, 위험, 대체 방식 제시
9. 사용자에게 보고: "이번에 정리할 것 / 다음으로 미룰 것 / 하지 않을 것"
   3분류로 제안. **사용자 승인 후에만 리팩터 배치 시작**

### D. 점검 기록
10. 이 파일 하단 점검 이력에 한 줄 추가 (날짜, 테스트 수, 번들 크기,
    QUAL 열림/닫힘 수, 결정 요약)

## 3. 리팩터 배치 규칙

- 리팩터는 기능 배치와 섞지 않는다 (한 커밋 = 리팩터만)
- 대체 전후로 같은 테스트가 통과해야 한다. 테스트가 없던 영역이면
  **먼저 현재 동작을 고정하는 테스트를 추가한 뒤** 대체한다
- 폴더 대이동 금지 원칙(CLAUDE.md §7) 유지 — 계약·진입점 먼저
- 대형 의존성 도입은 항상 별도 승인 (CLAUDE.md §8)
