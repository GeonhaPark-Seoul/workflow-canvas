# 범용 조작 생명주기 계약 v1

## 목적

캔버스가 구조를 보여주기만 하는 그림에서 실제 OS로 발전하려면 노드·파츠·연결선에서 안전한 조작을 시작할 수 있어야 한다. 하지만 Git 동기화, 배포, 주문 재시도, 물류 상태 변경, DB 마이그레이션을 각각 다른 방식으로 실행하면 사람, 자동화, 미래 AI마다 별도 우회로가 생긴다.

이 계약은 **의도를 받는 방법과 실제 권한 행사를 분리**한다.

- 사람은 캔버스 인터페이스를 직접 클릭한다.
- 결정론적 자동화는 승인된 조건이나 일정으로 시작한다.
- 미래 AI는 자연어를 계획으로 바꾸고 조작을 제안한다.

세 경로 모두 같은 계획, 승인, 실행, 검증, 감사, 중지, 재시도, 복구 계약을 통과한다. AI 연결은 필수가 아니며 직접 인터페이스가 첫 번째 정식 경로다.

## 미래 오케스트레이션 노드와의 관계

현재 단계에서 구현한 공통 안전 계층은 미래 계획 노드 중 다음 역할에 대응한다.

- A4 사람 승인 게이트: 실행 주체와 승인자를 구분하고 명시적 확인 문구를 검사한다.
- A6 실행 커넥터: 등록된 어댑터 ID와 동작 ID만 실행 대상으로 선언한다.
- A7 테스트·리뷰 게이트: 실행 완료 주장과 실행 후 성공 검증을 분리한다.
- A8 실시간 상태·로그: 상태 전이를 시간순 사건으로 남긴다.
- A9 결과물·감사 기록: 이전 사건 지문을 잇는 append-only 감사 사슬을 만든다.
- A10 중지·재시도·롤백: 실행 전 취소, 실행 중 중지 요청, 멱등 재시도, 선언된 복구를 구분한다.

A1 대화·명령 입력, A2 컨텍스트 조립, A3 의도·계획 엔진, A5 에이전트 배정은 아직 능동 실행하지 않는다. 나중에 구현해도 A4 이후의 계약을 우회할 수 없다.

## 정적 조작 정의

`shared/operationLifecycle.js`의 `normalizeOperationDefinition`은 어댑터와 시스템이 제공하는 조작 능력을 공통 형식으로 검증한다.

주요 필드는 다음과 같다.

- `availability`: 선언만 됨, 계획 중, 실행 가능, 비활성
- `allowedInitiators`: 직접 UI, 결정론적 자동화, AI 에이전트
- `authorizationPolicyIds`: 실제 권한·승인 정책
- `risk`, `sideEffect`: 위험도와 외부·데이터·권한 부작용
- `input`, `writeSet`, `excludes`: 입력 데이터 종류, 최대 쓰기 범위, 절대 제외 범위
- `execution`: 고정 실행 어댑터, 동작 ID, 실행 위치
- `timeoutMs`: 실행 시간 상한
- `idempotency`: 재전달·재시도의 중복 실행 방지 방식
- `verification`: 실행 후 검증 방식과 성공 조건
- `recovery`: 재시도 횟수, 복구 방식, 롤백 조작 참조

불완전한 조작은 `declared`나 `planned`로 표시할 수 있지만 `executable`이 될 수 없다. 즉 캔버스에 보인다는 사실이 실행 권한을 의미하지 않는다.

## 실행 상태

공통 상태는 다음 흐름을 사용한다.

```text
planned -> awaiting_approval -> approved -> queued -> running -> verifying -> succeeded
                              \-> rejected
approved/queued               \-> cancelled
running/verifying -> cancellation_requested -> cancelled
running/verifying -> failed -> retry queued 또는 recovery_pending
recovery_pending -> recovering -> recovered 또는 recovery_failed
```

불가능한 상태 전이는 코드에서 거부된다. 실행 어댑터는 자신이 시작한 실행 결과만 제출할 수 있고, `independent` 검증에서는 같은 주체가 자기 결과를 검증할 수 없다. 자동화와 AI는 조작을 시작할 수 있어도 사람 승인 역할을 사칭할 수 없다.

## 계획과 감사

범용 계획은 원본 입력 본문을 저장하지 않고 다음을 묶는다.

- 조작 정의 지문
- 정확한 대상과 Asset 원장 리비전
- 실행 전 상태 지문과 입력 지문
- 시작 주체와 경로
- 쓰기·제외 범위
- 위험, 승인, 시간 제한, 검증, 멱등성, 복구 선언
- 생성·만료 시각과 고유 nonce

런타임 사건은 순서 번호와 이전 사건 지문을 가진다. 이 지문 사슬은 코드 수준의 변경 감지 장치이며 단독으로 법적·암호학적 불변성을 주장하지 않는다. 실제 제품에서는 서버 서명과 DB append-only 정책, 백업, 외부 검증 가능한 감사 저장소를 함께 사용해야 한다.

## 현재 실제 기능에 연결된 부분

### 로컬 코드와 GitHub 동기화

- 표준 조작 정의와 실제 서명 계획이 같은 정의 지문을 사용한다.
- 웹 소유자 승인과 로컬 터미널 승인을 모두 유지한다.
- 일반 push와 fast-forward pull만 허용한다.
- 실행 후 다시 fetch해 브랜치, origin 지문, 작업 트리, ahead/behind, HEAD와 upstream 일치를 검사한다.
- 검증 결과가 없거나 맞지 않으면 에이전트가 성공을 주장해도 서버는 실패로 기록한다.

### 시스템 상태 불변 스냅샷

- 표준 조작 정의와 MCP 서명 계획이 같은 정의 지문을 사용한다.
- 소스 본문, 캔버스 본문, 이메일, 실제 자격증명 값을 제외한다.
- 스냅샷과 조작 감사 기록을 한 DB 트랜잭션으로 생성하고 반환 ID를 승인 계획과 대조한다.
- AI가 MCP 호출을 시작할 수 있지만 제품 소유자의 명시적 확인을 대신하지 못한다.

## 이번 단계에서 하지 않은 것

- 새 캔버스 버튼이나 범용 조작 패널을 추가하지 않았다.
- 모든 조작 실행을 저장하는 공통 DB 테이블로 기존 전용 테이블을 마이그레이션하지 않았다.
- 다자 승인, 배포·롤백, DB 마이그레이션, 권한 변경을 실행 가능하게 열지 않았다.
- AI 계획기, 장기 작업 큐, 24시간 에이전트를 시작하지 않았다.

다음 시스템을 가져올 때는 먼저 `declared` 또는 `planned` 조작으로 구조와 위험을 표현한다. 실제 실행을 열 때만 이 계약을 완성하고 골든 테스트를 통과시킨다.
