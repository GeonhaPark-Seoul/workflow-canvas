# Asset 원장 v3 정규형

## 무엇을 해결하는가

어댑터마다 캔버스 노드와 수정안을 직접 만들면 새 웹사이트, 물류 시스템, CRM을 붙일 때마다 검토 엔진과 UI까지 고쳐야 한다. Asset 원장은 공급자별 발견 결과와 캔버스 사이에 놓이는 공통 중간 형식이다.

Asset Build 처리 순서는 다음과 같다.

1. 어댑터가 허용된 원본에서 근거를 발견한다.
2. 어댑터가 발견 결과를 Asset 원장으로 정규화한다.
3. 공통 대조 엔진이 현재 캔버스와 Asset 원장을 비교한다.
4. 엔진이 추가·교체 가능한 변경만 지문이 붙은 수정안으로 만든다.
5. 사용자가 미리 보고 승인한 수정안만 기존 적용 엔진으로 전달한다.

Asset 원장은 소스 코드나 사용자 DB 본문을 복제하는 저장소가 아니다. 파일 경로, 자원 이름, 근거 지문, 제한된 설명처럼 시스템 구조를 확인하는 데 필요한 참조만 담는다.

## v3 기록 종류

### `source`

어떤 어댑터가 어느 스냅샷을 정규화했는지 기록한다. 어댑터 계약 버전, 어댑터 버전, 엔진 스키마 버전이 모두 포함된다.

### `entities`

`entities`는 Asset 목록의 wire 이름이다. API, 서비스, 데이터베이스, 사람, 배포 환경 같은 Asset 기록이 여기에 들어간다. 안정된 ID, 역할, 환경, 공급자, 외부 참조, 부모 Asset, 신뢰영역, 초기 배치 힌트를 가진다. 제품 엔진처럼 코드 안에 존재하지만 독립 배포 자원은 아닌 항목은 선택적으로 `logicalComponent`를 가진다.

### `logicalComponent`

중요한 코드 기능을 시스템 지도에서 설명하기 위한 논리 구성요소 메타데이터다. 내부 종류, 제품·기술 버전, 성숙도, 담당 Maintainer Agent ID, 입력·출력, 코드·테스트 근거와 호환성 계약을 기록한다. 이 필드가 있는 Asset은 독립 서버나 프로세스로 해석하지 않으며 런타임처럼 보이는 다른 필드가 있더라도 현실 수준은 `논리 구성`이다.

### `parts`

Asset이 제공하거나 사용하는 코드, 입력, 출력, 보기, 연결, 트리거, 키 참조다. 반드시 한 Asset에 속하며 필요하면 조작 계약을 참조한다.

### `relations`

두 Asset 또는 두 파츠 사이의 의미 있는 관계다. `reads`, `writes`, `calls`, `syncs_with` 같은 온톨로지 종류, 근거, 신뢰 게이트웨이, 캔버스 연결점 힌트를 가진다.

### `trustZones`와 `gateways`

로컬 기기, 사설망, 공개 클라우드, 외부 SaaS 같은 통제 경계를 표현한다. 서로 다른 신뢰영역의 Asset을 연결할 때는 실제 통과 지점인 게이트웨이가 반드시 있어야 한다. 게이트웨이의 방향, 노출 범위, 프로토콜, 데이터 종류, 인증·권한·암호화도 기록한다.

### `evidence`

파일, 설정, 커넥터 결과, 실행 기록, 문서의 참조와 지문이다. 실제 비밀 키나 토큰 값은 허용하지 않는다. 모든 Asset·파츠·관계·게이트웨이·조작은 존재하는 근거 ID만 참조할 수 있다.

### `dataClasses`

시스템을 오가는 정보의 종류다. 실제 행이나 본문이 아니라 `주문 레코드`, `Git 상태 메타데이터`, `조작 감사 메타데이터`처럼 의미·민감도·본문 포함 여부·보존 성격을 기록한다. 정책, 관측, 사건, 위협과 조작 입력은 이 ID를 공유하므로 어떤 정보가 어디에 쓰이는지 같은 언어로 추적할 수 있다.

### `policies`

누가 무엇을 읽고 쓰고 실행하거나 승인할 수 있는지 정하는 규칙이다. 허용·거부·승인 요구·한도 효과와 브라우저, 서버, DB, 커넥터 같은 실제 집행 지점을 분리한다. 정책 문구만 적혀 있고 집행 코드가 없는 상태도 숨기지 않는다.

### `observations`

특정 시점 또는 스냅샷에서 확인한 상태다. 선언, 코드 발견, 실제 관측, 런타임 검증, 오래됨, 모순, 알 수 없음을 구분한다. 관측 시각이 필요한 현실 수준에는 유효한 시각이 반드시 있어야 하며 만료된 값은 계속 LIVE로 남지 않는다.

### `events`

승인, 실행, 배포, 실패, 동기화, 롤백처럼 실제로 일어난 사건이다. 사건 종류, 대상, 행위자 종류, 발생 시각, 관련 데이터 종류와 근거를 가진다. 설정된 연결선 자체는 사건이 아니므로, 사건이나 반복 관측이 없으면 흐름 애니메이션을 켜지 않는다.

### `threats`와 `controls`

위협은 정상 연결선이 아니라 잠재적인 오용·침해 경로다. 시작과 도착, 데이터 종류, 가능성, 영향, 상태를 기록한다. 통제는 위협을 예방·탐지·제한·복구하는 인증, 권한, 검증, 격리, 감사 등의 장치다. 정상 데이터 흐름과 공격 가능성을 같은 관계로 섞지 않는다.

### `operations`

읽기·쓰기·실행 능력의 선언이다. v2 조작은 대상뿐 아니라 실행 가능 상태, 시작 주체, 정책, 위험도, 부작용, 입력 데이터 종류, 최대 쓰기 범위, 제외 범위, 실행 어댑터, 시간 제한, 멱등성, 검증, 재시도와 복구 선언을 가진다.

`declared` 또는 `planned` 조작은 구조에 나타낼 수 있지만 실행할 수 없다. `executable` 조작은 아래 조건을 모두 만족하지 않으면 빌드 자체가 거부된다.

- 허용된 시작 주체와 권한 정책이 있다.
- 부작용이 있으면 명시적 승인 문구와 최대 쓰기 범위가 있다.
- 임의 셸이 아니라 고정 실행 어댑터와 동작 ID가 있다.
- 실행 후 검증 어댑터와 성공 조건이 있다.
- 재시도 횟수가 2회 이상이면 멱등성 계약이 있다.
- 복구 가능 여부와 방식이 명시되어 있다.

## 참조 무결성과 보안 규칙

- 모든 기록 ID는 한 빌드 안에서 고유하고 재검사 후에도 안정적이어야 한다.
- 부모, 파츠 소유자, 관계 양 끝, 근거, 조작 대상, 정책, 데이터 종류, 통제, 위협 경로, 게이트웨이 참조가 실제로 존재해야 한다.
- 관계의 파츠 연결점은 해당 Asset에 실제로 속해야 한다.
- 서로 다른 신뢰영역을 잇는 관계는 양쪽 영역과 일치하는 게이트웨이 없이는 빌드 자체가 거부된다.
- 근거와 파츠 참조에는 실제 키·토큰·개인키 리터럴을 넣을 수 없다.
- 같은 입력은 정렬 순서와 무관하게 같은 정규형 지문을 만든다.
- 지원하지 않는 미래 스키마는 추측해 읽지 않고 명시적으로 거부한다.

## 사용자 데이터 소유권

공통 대조 엔진은 다음 시스템 계약만 비교한다.

- Asset의 종류와 의미 필드
- 정규화된 파츠 계약
- 관계의 종류·근거·파츠 연결점
- 신뢰영역과 게이트웨이 참조

다음 항목은 사용자가 소유하므로 자동으로 덮어쓰거나 삭제하지 않는다.

- 노드 위치와 크기
- 사용자가 작성한 메모와 알 수 없는 추가 필드
- 사용자가 직접 추가한 파츠·노드·연결선
- 기존 검토 결정

Asset 설명이 정규형과 다를 때도 현재 단계에서는 차이만 표시한다. 의미를 자동 교체하는 범용 작업은 아직 허용하지 않는다.

## 단계적 대조

부모 Asset이 없는 자식 Asset과 양 끝 Asset이 없는 관계는 즉시 추가하지 않는다.

1. 먼저 부모 또는 양 끝 Asset을 추가한다.
2. 다시 검사한다.
3. 파츠를 추가하거나 지문이 일치하는 기존 파츠를 교체한다.
4. 마지막으로 관계와 신뢰 게이트웨이를 추가한다.

이 순서로 한 수정안이 보이지 않는 선행 조건을 몰래 만들지 않으며, 각 단계가 현재 캔버스 상태를 다시 확인한다.

## 현재 참조 구현

- Workflow Canvas 시스템 지도: 실제 운영 구조와 제품·엔진 논리 구성층을 합쳐 54개 Asset, 9개 파츠, 56개 관계, 3개 데이터 종류, 3개 정책, 2개 실제 조작 계약, 2개 관측, 3개 통제, 1개 위협을 Asset 원장으로 정규화한다. 이 중 25개 Asset과 18개 포함 관계는 Engine registry에서 결정적으로 생성된다.
- 실제 Git 동기화와 불변 상태 스냅샷의 서명 계획은 각 표준 조작 정의의 지문과 묶인다. 조작 계약이 바뀌면 이전 미리보기는 실행되지 않는다.
- 주문 서비스 골든 픽스처: 공개 클라우드 API와 사설 데이터 영역, DB 게이트웨이, 파츠 연결 관계를 같은 엔진으로 대조한다.
- 주문 서비스는 DB Asset과 API 파츠를 먼저 적용한 뒤 관계를 적용하면 수정안 0개 상태가 된다.
- 이 과정에서 사용자가 옮긴 API 노드 위치·크기·메모와 직접 추가한 파츠가 보존된다.

## 마이그레이션

현재 스키마 버전은 `3`이다. 시험 단계의 v0 이름은 먼저 v1 이름으로 변환한 뒤 v2, v3으로 순차 전진 변환한다.

- `nodes` → `entities`
- `capabilities` → `parts`
- `connections` → `relations`
- `zones` → `trustZones`
- `actions` → `operations`

v1 조작은 ID, 대상, 근거를 보존한 `declared` 조작으로 승격한다. 비어 있던 정책·관측·사건·통제·위협 목록을 추가하지만, 과거 선언을 근거 없이 실행 가능하다고 승격하지 않는다. v2 `entities` 기록은 기존 의미와 ID를 보존한 채 `logicalComponent: null`을 추가해 v3으로 승격한다. 과거 운영 자원을 근거 없이 논리 엔진으로 분류하지 않으며 지원하지 않는 미래 버전은 자동 추정하지 않는다.

## 아직 남은 범위

- Asset 의미 필드의 안전한 부분 교체 계약
- 실제 두 번째 언어·프레임워크 저장소 어댑터
- 대규모 빌드의 증분 대조와 성능 한도
- 정책·관측·사건·통제·위협을 캔버스 필터와 공격 경로 오버레이로 구체화
- 범용 조작 실행 이력을 저장할 공통 DB 사건 테이블과 다자 승인 집계
