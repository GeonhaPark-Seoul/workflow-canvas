# Twin Adapter Contract v1 (레거시 현행 코드명)

## 이 계약이 필요한 이유

Workflow Canvas 전용 탐색 코드를 계속 엔진 안에 추가하면 다른 웹사이트, 물류 시스템, CRM, 가계부를 가져올 때마다 앱 전체를 고쳐야 한다. `Twin Adapter Contract`는 분해 전까지 새 이름을 만들지 않고 유지하는 레거시 현행 코드명이다. 이 계약은 각 시스템의 차이를 Adapter 안에 가두고, 검토·권한·지도 적용·조작 안전성에는 같은 Engine을 재사용하기 위한 경계다.

사용자 화면에서는 이 과정을 **시스템 가져오기**라고 부른다. 내부에서는 어댑터가 근거를 읽고, 엔진이 검증한 검토 결과를 만든다.

## 현재 v1이 보장하는 것

- 모든 어댑터는 고유한 `id`, 자체 버전, 계약 버전, 호환 엔진 스키마 범위를 선언한다.
- 어댑터가 읽는 데이터 종류, 원본 밖으로 나가는지 여부, 본문 포함 여부, 민감도를 미리 선언한다.
- 필요한 읽기·쓰기·실행 권한과 범위를 미리 선언한다.
- 지원한다고 선언한 인터페이스가 실행 모듈에 실제로 존재해야 한다.
- 등록 시 본 설명서와 나중에 불러온 실행 모듈의 설명서 지문이 같아야 한다.
- 어댑터에는 원본 캔버스가 아니라 동결된 복제 스냅샷만 전달한다.
- 어댑터가 반환한 검토 결과의 출처, 버전, 항목 식별자, 지문, 중복을 엔진이 다시 검사한다.
- 어댑터는 캔버스 JSON을 직접 저장하거나 검토 절차를 우회하지 않는다.

## 필수 인터페이스

### `describe`

버전, 지원 시스템, 데이터 범위, 권한, 기능, 조작 능력을 반환한다. 지연 로딩 전 등록 정보와 실행 모듈의 정보가 정확히 같아야 한다.

### `canInspect`

현재 캔버스가 해당 어댑터의 시스템인지 읽기 전용으로 판별한다.

### `inspect`

동결된 캔버스 스냅샷을 읽고 공통 Asset 검토 결과를 반환한다. 직접 쓰기는 허용하지 않는다.

## 선택 인터페이스

- `discover`: 공급자별 원본 근거 수집
- `normalize`: 원본 근거를 범용 Asset 원장 기록으로 변환
- `reconcile`: 안정된 식별자와 변경 수정안 생성
- `planOperation`: 조작 대상, 쓰기 범위, 위험, 만료, 복구 계획 생성
- `executeOperation`: 허용 목록 안의 조작 실행
- `verifyOperation`: 실행 결과를 독립적으로 검증
- `rollbackOperation`: 지원되는 복구 실행 또는 복구 불가 선언
- `redact`: 외부 전송 데이터와 비밀값 차단
- `migrate`: 어댑터 상태와 스키마의 전진 마이그레이션

현재 v1은 `describe`, `canInspect`, `inspect`만 모든 어댑터의 필수 인터페이스다. Workflow Canvas 참조 어댑터는 C2부터 `normalize`와 `reconcile`도 구현한다. 구현하지 않은 인터페이스를 지원한다고 선언할 수 없다.

## 새 시스템을 붙이는 절차

1. 어댑터 설명서에 시스템 종류, 데이터 범위, 권한을 선언한다.
2. 해당 시스템을 식별하는 최소 서명을 `canInspect`에 구현한다.
3. 읽기 전용 근거를 공통 검토 결과로 변환한다.
4. 고정 입력과 예상 출력으로 골든 픽스처를 만든다.
5. 같은 입력이 같은 지문을 만드는지 검사한다.
6. 사용자의 수동 배치, 메모, 검토 결정이 재검사 후에도 보존되는지 검사한다.
7. 범용 레지스트리에 등록한다. 검토 UI나 캔버스 엔진은 수정하지 않는다.

## 현재 구현과 다음 단계

Asset 원장 v3가 Asset, 파츠, 관계, 신뢰영역, 게이트웨이, 데이터 종류, 정책, 근거, 관측, 사건, 조작, 위협과 통제를 공급자 중립 기록으로 묶는다. v3은 제품 엔진처럼 독립 배포 자원이 아닌 논리 구성요소의 버전, 성숙도, 입출력과 코드·테스트 근거도 보존한다. 공통 대조 엔진은 현재 캔버스와 정규형을 비교해 안전한 추가·파츠 교체·관계 교체 수정안을 만든다.

실행 가능한 조작은 `shared/operationLifecycle.js`의 공통 안전 조건을 만족해야 한다. 직접 인터페이스, 결정론적 자동화, 미래 AI는 서로 다른 어댑터 우회로를 만들지 않고 같은 조작 정의와 상태 전이를 사용한다. 현재 Workflow Canvas의 Git 동기화와 불변 상태 스냅샷은 실제 서명 계획을 각 표준 조작 정의 지문에 묶는다.

설명서의 `operationCapabilities`는 해당 시스템에서 발견해 지도에 설명할 수 있는 조작 ID 목록이다. 이 목록만으로 어댑터에 실행 권한이 생기지 않으며, 실제 실행은 별도 `planOperation`·`executeOperation` 구현과 서버 권한, 범용 조작 계약을 모두 통과해야 한다.

현재 참조 구현은 다음 순서를 사용한다.

1. 어댑터가 근거를 발견한다.
2. 범용 Asset 원장 기록으로 정규화한다.
3. 엔진이 현재 캔버스와 대조한다.
4. 엔진이 공통 수정안을 만든다.

다음 단계에서는 Python/FastAPI처럼 구조가 다른 실제 두 번째 저장소를 같은 계약으로 가져와 일반성을 검증한다. 그 과정에서 Workflow Canvas에 남아 있는 공급자별 발견·설명 코드를 해당 어댑터 안으로 더 얇게 이동하고, 새 시스템의 조작은 먼저 `declared` 또는 `planned` 상태로만 가져온다.
