> 이 문서는 과거의 고정 L1~L4 및 L1~L5 시스템 지도 계약을 폐기하고 다시 시작하는

> 하나씩 검토한 뒤 설계하고 구현한다.

## 1. 제품 목표

소프트웨어 개발업 전체를 코드와 도구 목록이 아니라 탐색하고 조작할 수 있는 시각적
세계로 만든다. 사용자는 복잡한 코드를 모두 읽지 않아도 다음을 할 수 있어야 한다.

- 무엇이 실제로 존재하는지 본다.
- 코드·데이터·요청이 어디에 있고 어디로 움직이는지 이해한다.
- 현재 상태와 문제의 근거를 확인한다.
- 권한과 승인 경계 안에서 실제 개발 작업을 실행한다.
- 필요하면 개발자용 상세 화면까지 내려간다.

`로컬 저장소 → GitHub → Vercel`은 가능한 흐름 하나일 뿐이며 제품 전체의 기준
구조가 아니다.

## 2. 확정된 노드 원칙

> **노드는 코드·데이터·요청이 머물거나 실행되는 지속적인 장소 또는 독립
> 시스템이다.**

화면에 무언가를 노드로 올리기 전에 다음을 확인한다.

1. 한 번의 요청·작업·배포가 끝나도 그 대상의 정체성이 남는가?
2. 코드·데이터·요청이 그 안에 머물거나 실행되는가, 또는 다른 것과 독립된
   시스템인가?
3. 안정적인 식별자·주소·연결 정보·관리 표면 중 하나 이상을 가질 수 있는가?

세 질문을 통과하지 못하면 기본적으로 노드가 아니다.

예를 들어 저장소, 데이터베이스, 실행 서비스, 패키지 저장소, 외부 SaaS는 노드가
될 수 있다. 반대로 Pipeline, Job, Build, Release, Deployment, Request, Commit,
PR은 대체로 흐름·실행·사건·버전이므로 노드보다 다른 UI가 적합하다. 이름만 보고
일괄 판정하지 않고 실제 대상의 지속성과 독립성을 확인한다.

## 3. Asset과 화면 노드

### 3.1 Canonical Asset

Canonical Asset은 화면 배치와 무관한 실제 대상의 원장 기록이다. 전체 Asset 원장에는
사람·결정·문서·사건처럼 관리할 가치가 있지만 노드가 아닌 대상도 있을 수 있다.
View Node는 그중 §2를 통과한 Asset만 참조한다.

```text
CanonicalAsset
- assetId
- nodeForm: place | independent-system | null
- provider / accountId / externalId
- aliases: URL, 경로, 과거 이름
- capabilities
- owner / permissions / trustZone / lifecycle
- current observations / evidence / realityLevel
```

`nodeForm`이 `null`이면 다른 UI로 표현하며 노드로 만들지 않는다. URL·이름·로컬
경로는 바뀔 수 있으므로 내부 `assetId`와 동일하게 취급하지 않는다.
로컬 저장소와 원격 저장소처럼 비슷해 보여도 서로 다른 장소라면 각각 별도 Asset이고
`clone_of`, `syncs_with` 같은 관계로 연결한다.

### 3.2 View Node

같은 Canonical Asset을 여러 화면에서 각각 필요한 모습으로 표현할 수 있다.

```text
ViewNode
- viewNodeId
- viewId
- assetId
- position / size / collapsed
- visibleParts
- local label / group / emphasis
```

동기화되는 것:

- 실제 대상의 식별과 연결 상태
- 관측된 상태·버전·오류·근거
- 소유권·권한·신뢰 경계
- 다른 Asset과의 실제 관계

화면별로 독립인 것:

- 위치·크기·접힘
- 이 화면에서만 보이는 파츠
- 그룹·강조·임시 별칭
- 카메라와 LOD 상태

따라서 한 Asset의 실제 상태가 바뀌면 모든 화면의 노드가 함께 갱신되지만, 한
화면의 배치를 바꿔도 다른 화면의 배치는 움직이지 않는다.

## 4. 노드가 아닌 것을 보여주는 문법

모든 정보를 노드로 만들지 않는다.

| UI 형태 | 적합한 대상 |
|---|---|
| **파츠·패널** | 노드 내부의 기능, 파일 계층, 스키마, 설정, 버전, 상태, 로그 |
| **그룹·영역** | 환경, 계정, 팀, 제품 경계, 신뢰영역처럼 여러 노드를 묶는 관점 |
| **연결선** | 두 지속 실체 사이의 비교적 안정적인 관계 |
| **흐름·경로** | CI/CD, 요청 처리, 데이터 이동, 승인 과정 |
| **Run·Event 타임라인** | Job, Build, Test, Release, Deployment, Incident, Commit |
| **Actor·Presence 표시** | 사람, AI 에이전트, 현재 편집자, 승인자 |
| **오버레이** | 보안, 상태, 변경, 비용, 소유권, 위험, 근거 수준 |
| **목록·원장·드릴다운** | 수가 많아 지도 노드로 펼치면 읽을 수 없는 세부 항목 |

같은 대상도 화면의 질문에 따라 파츠·그룹·오버레이 중 다른 형태가 적합할 수 있다.
단, 노드의 자격만큼은 §2의 확정 원칙을 바꾸지 않는다.

## 5. 화면/레이어의 정의

레이어는 현실을 나누는 공식 온톨로지가 아니라 특정 작업을 하기 위한 화면이다.

```text
View
= 질문
+ 표시할 Asset과 관계
+ 그룹 규칙
+ 노드 파츠
+ 흐름과 오버레이
+ 허용된 관측·실행 동작
```

- 공식 층수와 순서를 두지 않는다.
- 모든 프로젝트에 같은 화면을 강요하지 않는다.
- 비개발자용 화면과 개발자용 화면이 동시에 존재할 수 있다.
- 사용자 요청으로 새 화면을 그대로 추가할 수 있다.
- 화면을 추가하기 위해 Asset을 복제하지 않는다.
- 화면 간 공통 Asset은 `assetId`로 동기화한다.

## 5A. 내부 화면(포털) 패턴 — 노드 안에 또 다른 화면

파츠·배지·목록만으로 "이 대상이 어떤 덩어리들로 이루어졌고 서로 어떻게
영향을 주는지"를 한눈에 보여주기 어려운 경우가 있다(예: 저장소 하나가 여러
컴포넌트로 나뉘는 것). 파츠 패널은 목록이지 공간 배치가 아니라서, 클릭해서
세부 패널을 열어야만 구조가 보이면 "코드를 안 읽고도 한눈에 이해한다"는 목적이
후퇴한다.

**이건 데이터 모델 문제가 아니라 표현(UI) 문제였다.** Canonical Asset/View
Node 구조는 처음부터 이걸 지원한다 — 그냥 새 View를 하나 더 만들면 된다.

**패턴:** 특정 Asset이 내부에 공간적 구조(그룹·관계)를 가질 만큼 복잡하면,
그 Asset의 "열기" 동작이 **내부 화면**이라는 전용 View로 이동시킨다.

- 내부 화면은 §5의 View 정의를 그대로 따르는 진짜 View다. 새 개념이 아니라
  "이 Asset에 속한 것들"로 범위가 자동 필터링된 View일 뿐이다.
- 진입점(부모 Asset의 "열기" 동작) → 내부 화면(포함 관계로 범위가 좁혀진
  View, 그 안에서 컴포넌트 등은 §4의 그룹 문법으로 표현) → 나가기(부모 화면
  복귀)의 3단 계약을 갖는다.
- **재사용 가능한 일반 옵션이다.** 저장소 내부(컴포넌트 그룹)뿐 아니라, 데이터
  저장소 내부(테이블을 도메인별로 그룹), 호스팅 계정 내부(배포를 서비스별로
  그룹) 등 비슷한 문제가 생기는 어떤 노드에도 같은 패턴을 쓴다. 새로 등장하는
  화면마다 이 옵션을 쓸지 §4의 단순 파츠/배지로 충분한지 §9 절차의 6단계에서
  매번 판단한다.

## 6. 화면 후보 레지스트리

아래는 구현 후보 목록이지 확정 레이어 계약이나 구현 순서가 아니다. 각 후보는
사용자와 한 번에 하나씩 검토한다.

| 화면 후보 | 핵심 질문 |
|---|---|
| 개발 실체 인벤토리 | 이 소프트웨어를 이루는 실제 장소와 독립 시스템은 무엇인가? |
| 코드 세계 | 코드는 어디에 있고 내부 구조와 영향 관계는 어떻게 생겼는가? |
| 개발 작업공간 | 사람과 AI는 어느 작업공간에서 무엇을 바꾸고 있는가? |
| 소스 협업 | 변경이 어떻게 제안·검토·병합되는가? |
| CI/CD 전달 | 변경이 어떤 검사와 전달 과정을 거쳐 실행 가능한 결과가 되는가? |
| 테스트·품질 | 무엇을 어떻게 검증하며 현재 빈 구간은 어디인가? |
| 릴리스·환경 | 각 환경에 어떤 버전과 설정이 반영돼 있는가? |
| 런타임 토폴로지 | 실제로 살아 있는 서비스와 실행 장소는 무엇인가? |
| 요청 흐름 | 사용자 요청이 어디를 지나 어디에서 실행되는가? |
| 데이터 세계 | 데이터가 어디에 저장되고 변환·이동·삭제되는가? |
| API·통합 | 내부·외부 시스템이 어떤 계약과 경계로 연결되는가? |
| 의존성·공급망 | 어떤 패키지·이미지·외부 산출물에 의존하는가? |
| 운영·장애 | 무엇이 아프고 영향 범위와 복구 경로는 무엇인가? |
| 보안·신뢰 | 어떤 주체가 어떤 경계를 넘어 무엇에 접근할 수 있는가? |
| 제품 표면 | 사용자는 어떤 실행 중인 제품과 접점에 닿는가? |
| 의사결정·작업 | 요구·설계·작업·코드 변경은 어떻게 이어지는가? |
| 소유권·비용 | 누가 책임지고 어떤 시스템에서 비용이 발생하는가? |

후보에 없는 화면도 사용자의 질문이 명확하면 추가한다.

## 7. 보안의 위치

보안 문제는 CI/CD Pipeline 안에서만 생기지 않는다. 소스 코드, 의존성, 자격증명,
권한, 데이터, 런타임, 네트워크, 클라이언트, 외부 연동, 개발 작업공간 모두에서
생길 수 있다.

따라서 보안은 다음 두 형태를 함께 가진다.

1. 여러 화면 위에 켜는 보안·신뢰 오버레이
2. 권한과 공격 표면을 깊게 조사하는 전용 보안 화면

취약점과 보안 진단 결과는 보통 노드가 아니라 대상 Asset에 붙는 finding, 배지,
오버레이, 타임라인이다. 보안 스캐너나 IAM 서비스처럼 독립 시스템인 대상만 노드가
될 수 있다.

## 8. 3D 확장

3D는 별도 데이터 모델이 아니라 같은 Asset과 View 데이터를 그리는 렌더러다.
동일한 Asset이 여러 화면에 등장할 때, 3D에서는 여러 층을 가로지르는 직육면체,
원기둥, 축 같은 하나의 큰 형상으로 연결해 표현할 수 있다. 구체적인 높이와 형상은
2D 화면과 동기화 계약을 검증한 뒤 정한다.

## 9. 한 화면씩 검토하는 절차

1. 그 화면이 답해야 할 사용자 질문을 한 문장으로 정한다.
2. 왜 필요한지와 비개발자·개발자에게 각각 어떤 가치가 있는지 설명한다.
3. 관측할 Work와 Workflow를 정한다.
4. 실행할 수 있는 동작과 승인 경계를 정한다.
5. 노드 후보를 §2의 기준으로 하나씩 판정한다.
6. 나머지 대상을 파츠·그룹·흐름·이벤트·오버레이로 배치한다.
7. 필요한 데이터 근거와 Connector를 확인한다.
8. 사용자 승인 뒤에만 그 화면의 시각안과 구현을 진행한다.

다음 화면으로 넘어가기 전에 현재 화면의 목적·노드·워크플로·표현 문법을 사용자와
확정한다.

## 9A. 적용 사례 — 코드 세계 화면 (§9 절차 1~6단계 완료, 설계 검토 대기)

첫 화면으로 "로컬→GitHub→Vercel" 흐름이 아니라 **코드 세계**를 선택했다 —
제품의 본래 목표("파일과 코드 줄을 읽지 않고도 이 소프트웨어가 어떤
덩어리들로 이루어졌고 무엇이 무엇에 영향을 주는지 이해한다")에 가장 가깝다.

**1. 질문:** 코드는 어디에 있고 내부 구조와 영향 관계는 어떻게 생겼는가?

**2. 가치:** 비개발자는 파일을 열지 않고 구조를 이해하고, 개발자는 영향
범위·의존 관계를 추적한다.

**3. 관측할 Work:** 코드 위치, 로컬·원격 차이, 내부 의존 관계, 변경 영향
범위, 현재 편집 활동과 근거(→ `LIVE_EDIT_OBSERVATION.md`와 연동).

**4. 실행할 Work와 승인 경계:**

| 동작 | 실행 방식 | 승인 |
|---|---|---|
| 저장소 열기 (내부 화면 진입, §5A) | 노코드 | 완전 자동 |
| 재스캔 | 예스코드 | 완전 자동 |
| 코드 영역 설명 (결정적) | 예스코드 | 완전 자동 |
| 영향 추적 | 예스코드 | 완전 자동 |
| 변경 비교 | 예스코드 | 완전 자동 |
| 수정 제안 | 노코드 (SL-3 경로) | 웹 승인 |
| 테스트 실행 | 예스코드 | 완전 자동 — 부작용 없는 읽기성 작업이라 Engine 0.x "읽기·대조" 범위 안. 결과를 Asset 관측으로 기록하는 것까지 포함, 별도 승인 불필요 |

**5. 노드 판정 (§2 세 질문 기준):**

| 후보 | 판정 | 비고 |
|---|---|---|
| 로컬 코드 저장소 | ✅ 노드 | 경로가 바뀌어도 `assetId`는 유지 |
| 원격 소스 저장소 | ✅ 노드 | 로컬과 별개 실체, `clone_of`/`syncs_with`로 연결 |
| **패키지 저장소** | ✅ 노드, **조건부** | **외부 레지스트리**(npm·PyPI 등)가 맞다. 단 "레지스트리 자체"가 아니라 "이 소프트웨어가 그 레지스트리에 게시하는 패키지"를 가리킨다 — 원격 소스 저장소와 같은 패턴(외부 서비스 안에서 이 프로젝트가 차지하는 몫). 패키지를 게시하지 않는 소프트웨어(대다수의 바이브 코더 웹앱)엔 이 노드가 없다. 외부 서비스이므로 공급망 보안(의존성 취약점 등, `SYSTEM_MAP_BLUEPRINT` 이전 버전에서 발견한 미해결 갭)이 훗날 붙을 자리이기도 하다 |
| 지속적인 개발 환경 | ✅ 노드, 조건부 | Codespaces류. 로컬 개발만 하면 없음 |
| 독립된 코드 분석 시스템 | ✅ 노드, 조건부 | Graphify 등 외부 연동 시에만 |

**6. 노드 아닌 것 배치:**

- 디렉터리·모듈·파일 → 저장소의 **내부 화면**(§5A)에서 컴포넌트별 **그룹**으로
  배치. 저장소 노드를 열면(포털) 이 내부 화면으로 이동 — 클릭 한 번으로
  구조가 보이므로 "한눈에 이해" 목적을 만족한다. 파츠 패널 안에 숨기지 않는다.
- 서브시스템/컴포넌트 → 위 내부 화면의 그룹 경계. MASTER §2.1의 기존 정의
  ("컴포넌트 = 정제된 분류, 코드에 반영 안 됨")와 일치 — 별도 Asset으로 만들지
  않는 게 오히려 원래 정의에 더 정직하다.
- Branch·Commit·PR·Diff·Test Run → 저장소(또는 관련 노드)의 타임라인·흐름.
- 현재 편집 활동(사람·AI) → Actor·Presence 표시. `LIVE_EDIT_OBSERVATION.md`가
  이 화면의 첫 실전 적용 대상이 된다.

**7. 데이터 근거·Connector (기존 자산 재사용, 신규 없음):** 코드 Asset
계층(파일→모듈→컴포넌트 해석, SL-1에 이미 구현)이 내부 화면의 그룹 배치
근거. 로컬 커넥터(로컬 저장소 상태), 기존 GitHub 연동(원격 저장소), 패키지
레지스트리 Connector(있을 때만, 신규 검토 필요).

**8. 승인 대기.** 위 1~7은 사용자·Claude Code 검토로 확정. 시각안·구현은
Codex 설계 검토 라운드 이후 진행.

## 10. 폐기된 계약

- 보호된 공식 L1~L4 프리셋
- 고정 L1~L5 교체안
- 시스템 종류를 공식 층에 자동 배정하는 표
- 추상화 깊이 하나만을 모든 화면의 고정 Z축으로 삼는 규칙
- 하나의 노드 표현을 여러 화면에서 공유해야 한다는 가정

과거 구현에서 재사용하는 것은 사용자 정의 화면, 투영, 포털, 관측, redaction 같은
일반 기능뿐이다.
