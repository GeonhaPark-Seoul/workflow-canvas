# 오픈소스·표준 우선 정책

## 목적

Workflow Canvas OS의 차별점은 이미 잘 해결된 기반 기술을 다시 만드는 데 있지 않다. 차별점은 서로 다른 소프트웨어를 근거 기반 디지털 트윈으로 만들고, 비개발자가 이해하고 제한적으로 조작하게 하는 제품 경험에 있다. 파서, 그래프 배치, 권한 판정, 작업 실행, 프로토콜처럼 일반적으로 검증된 기반 영역은 새 코드를 만들기 전에 표준과 성숙한 오픈소스를 먼저 비교한다.

이 정책은 오픈소스를 무조건 도입하라는 뜻이 아니다. 작은 자체 구현이 더 명확하고 안전할 수 있으며, 새 의존성은 번들 크기, 공급망, 라이선스, 운영 복잡도와 마이그레이션 비용을 만든다.

## 새 기반 기능의 필수 절차

1. 해결하려는 사용자 문제와 필요한 계약을 먼저 적는다.
2. 관련 표준, 브라우저·Node 기본 기능, 현재 의존성, 검증된 오픈소스 후보를 조사한다.
3. 자체 구현 유지와 후보 도입을 정확성, 성능, 보안, 유지보수, 번들·운영 비용, 라이선스, 교체 가능성으로 비교한다.
4. 결과를 [`DEPENDENCY_DECISIONS.md`](DEPENDENCY_DECISIONS.md)에 기록한다.
5. 대형 라이브러리, 외부 권한 서비스, 상주 Worker 또는 새 데이터 저장소는 사용자에게 효과와 비용을 설명하고 명시적 승인을 받는다.
6. 승인된 직접 의존성만 [`dependency-registry.json`](dependency-registry.json), `package.json`, lockfile과 [`../../THIRD_PARTY_NOTICES.md`](../../THIRD_PARTY_NOTICES.md)에 함께 반영한다.
7. 테스트와 빌드에서 `npm run governance:check`를 통과한다.

## 도입 판단 기준

- 실제 제품 오류나 유지보수 비용을 줄이는가
- 현재 지원 범위에서 필요한 기능인가
- 보안 경계와 데이터 이동을 더 명확하게 만드는가
- 결정적 출력과 안정된 버전을 제공하는가
- 브라우저 번들 또는 서버 운영 비용이 수용 가능한가
- 프로젝트와 배포 방식에 맞는 라이선스인가
- 나중에 다른 구현으로 교체할 수 있는 얇은 계약 뒤에 둘 수 있는가
- 테스트 fixture와 마이그레이션 계획을 만들 수 있는가

후보가 유명하다는 이유만으로 도입하지 않는다. 현재 코드가 작고 검증되어 있으며 제품 문제가 없다면 `유지`도 정식 결정이다.

## 현재 승인 경계

- 기존 직접 의존성은 초기 레지스트리에 `approved-existing`으로 기록한다.
- `elkjs`, Dagre, OpenFGA, SpiceDB, Tree-sitter, Temporal 같은 항목은 비교 후보일 뿐 승인된 의존성이 아니다.
- 후보를 실제 코드에 import하거나 외부 서비스를 배포하려면 결정 상태와 사용자 승인을 먼저 갱신한다.
- 새 의존성 없이 계약·테스트·문서만 추가하는 작업은 이 경계 안에서 진행할 수 있다.

## SBOM과 출시 검사

- `npm run sbom:generate`는 현재 lockfile의 CycloneDX SBOM을 표준 출력으로 생성한다.
- 공개 출시 전 CI는 각 배포 커밋의 SBOM, 직접·전이 의존성 라이선스 보고서와 취약점 결과를 보존해야 한다.
- 현재 `governance:check`는 직접 의존성, 잠금 버전, 라이선스와 고지 누락을 차단한다. 취약점 데이터베이스 조회, 전이 라이선스 법률 검토, CI artifact 보존은 `OPS-006`의 남은 출시 부채다.
- 보안 검사 결과가 없다는 사실을 안전성 증명으로 표현하지 않는다.

## 예외

긴급 보안 수정처럼 절차를 먼저 끝낼 수 없는 경우에도 예외 소유자, 범위, 이유, 만료일과 후속 검토를 dependency registry에 기록해야 한다. 영구적인 구두 예외는 허용하지 않는다.
