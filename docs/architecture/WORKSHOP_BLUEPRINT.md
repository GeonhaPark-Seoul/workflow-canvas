> 웹·앱 개발업의 전 과정(기획→개발→배포→운영→수익→성장→고객→거버넌스)이

## 1. 확정된 제품 정체 (라운드 1)

**작업장(Workshop)이다.** 개발업의 일 자체가 제품 안에서 일어난다. 다만:

- **AI가 모든 단계에 관여할 수 있다.** 각 단계의 담당 AI 에이전트를 사용자가
  배정하고, 대화할 수 있고, 사용자가 보는 것을 AI도 본다.
- **실행 방식은 배정이다.** 각 일에 수동(사람) / 예스코드(결정적 자동화 — AI 토큰
  절약) / 노코드(AI 대행)를 배정한다. `FOUNDRY_MODEL.md` §10의 2축 라벨이 문서용
  분류를 넘어 **런타임 배정표**로 승격됐다.
- **병렬이 기본이다.** 어떤 일은 기획, 어떤 일은 개발, 어떤 일은 배포 단계에 동시에
  존재하며 이를 한눈에 본다.
- 기존 MASTER §3 ①의 "관제실" 정체성은 작업장의 부분집합이 된다 (관제 = 작업장의
  보기·승인 기능). **MASTER §3 개정 필요 — 반영 전까지 이 문서가 선행 기획.**

첫 구현 구간: **A 기획**. 첫 사용자: **나 자신** (도그푸딩 — 사용자+Claude Code 기획,
Codex 구현, 사용자 승인이라는 현재 협업 자체가 첫 번째 개발업 워크플로우).

## 2. 확정된 공간 구조 (라운드 2·3)

**두 표면(surface) + 한 진입점:**

```
[캔버스] ─ 실체의 지도 (기존 제품 보존)
   DB · 로컬 저장소 · Vercel · GitHub · 모듈 = 시스템 지도 그대로
   + 관제 노드: 누르면 ↓ 가 캔버스 위 대형 화면으로 열림

[보드] ─ 워크플로우 오케스트레이션 (새로 만듦)
   상단 가로: A~H 8단계 열 (기획/개발/배포/운영/수익/성장/고객/거버넌스)
   맨 왼쪽 열: 백로그 (아이디어·목표 목록)
   카드 = 목표 하나 (Workflow Run) — 게이트 승인을 받으며 오른쪽으로 이동
   카드 열면: 그 목표의 대화방 + 담당 배정 + 참조 실체
```

- 기존 캔버스는 버리지 않는다. 3년치 엔진(동기화·공유·권한·redaction)은 실체
  지도로 그대로 산다. "뒤집기"의 실체는 **오케스트레이션 표면을 얹고 제품 중심을
  옮기는 것**.
- 파운드리 문법: 캔버스 = 온톨로지 뷰, 보드 = 액션/워크플로우 뷰.
- 캔버스 쪽 시각화 세부(토큰 표시 등)는 나중에 논의한다 (사용자 결정).

## 3. 확정된 보드 규칙 (라운드 3)

| 항목 | 결정 |
|---|---|
| 열 구성 | **A~H 8열 전부** + 왼쪽 백로그. E~H는 당분간 빈 열 — 사업의 나머지 절반이 보이는 정직한 지도로 의도적 유지 |
| 카드 단위 | **목표 하나** (예: "용어 정리 패스") = 보드의 접이식 1급 항목. 그 아래 **작업 트리**가 있고 각 작업이 자기 단계(열)에 위치 — 같은 목표의 작업들이 서로 다른 단계에 동시 존재한다. 작업 중 발견은 다른 단계에 새 작업으로 파생되며 파생 관계(어느 작업이 낳았나)를 가지로 표시한다(기존 Relation 문법 재사용, 작업도 Asset). 거대한 트리는 접기/펴기로 관리. MASTER Workflow 계층(Definition→Run→Work Item)의 본체 승격 |
| 단계 전환 | **게이트 승인, 기본값 전부 수동, 강제는 전부 소프트.** 담당(AI·자동화)이 완료 보고 → 승인 대기 표시 → 사용자 확인 후 이동. 산출물이 없어도 경고만 하고 통과 가능하되 **강행 기록이 이력에 남는다** — 품질 문제 발생 시 어느 게이트가 헐거웠는지 추적, 필요 시 열별 하드 전환은 나중에. 열별 "자동 통과"도 사용 패턴이 보인 뒤 추가 (§10 승인 루트 재사용) |
| 완료 정의 | **목표마다 종착 열 지정.** 카드 생성 시 "어디까지 가면 완료"를 정한다 (기획만으로 끝나는 조사 목표, 배포까지 가는 기능 목표, 운영에 상주하는 감시 목표). 종착 열 통과 승인 = 완료·아카이브 |
| 대화 | **대화 트리.** 목표·작업의 대화는 가지 구조로 저장되고, 상위/하위 가지를 오가며 어느 가지에서든 이어갈 수 있다(가지에서 이어가기 = 그 지점 맥락으로 세션 재개). 참조 실체·단계 맥락 자동 전달은 AI Context Gate의 Planning Context Pack 확장. 대화 기록 = 그 목표의 근거 |
| 카드 생성 | **어디서든 + 백로그 집결.** 보드 직접 작성 / 카드 대화방에서 파생 / 캔버스 실체에서 생성("이 모듈에 문제" → 일감화) / 운영 관측(장애·stale)에서 자동 제안. 자동 생성분은 dimmed(제안 상태)로 구분 — 주장과 사실 분리 원칙 유지 |
| 거처 | **캔버스 위 관제 노드 → 대형 화면.** 별도 페이지·전역 rail이 아니라 캔버스 공간 UX 원칙(기능은 엔티티에 붙는다)대로 |

## 4. AI 에이전트의 실체 (라운드 2 — Claude Code 판단 위임분)

**CLI 에이전트 연결로 시작한다** (Claude Code·Codex 등 사용자가 이미 쓰는 로컬
에이전트를 로컬 커넥터로 연결).

- 근거: (a) 실제 개발 노동(코드·커밋·배포)을 이미 할 줄 아는 실행 주체, (b) 사용자
  Mac에서 사용자 권한·계정으로 돌아 최소 권한 원칙과 로컬 커넥터 구조에 맞음,
  (c) API 직접 호출 방식은 실행기·큐·샌드박스·감사를 새로 지어야 하고 상시형 AI
  금지 원칙과 충돌, (d) 도그푸딩 대상(우리)이 이미 CLI 에이전트 2대 편성.
- **에이전트 배정은 Adapter 계약 뒤에 둔다** — 나중에 API 내장형·다른 provider로
  교체 가능해야 함. "나중에 바꿔도 되는가"의 답이 Adapter 문법 그 자체.

## 4A. 컨텍스트 4층 모델과 세션 규칙 (라운드 5 확정)

```
프로젝트 컨텍스트  ← Project Master. 느리게 변하고 사용자가 소유
카드 컨텍스트      ← 목표의 여권: 산출물·결정·대화 요약·참조 실체가 단계를 여행하며 축적
세션 컨텍스트      ← 작업당 하나의 세션 계보. 제한 팩을 받고 완료 보고를 남긴다
단계 컨텍스트      ← 열의 계약(카드와 직교): 입력·산출물 형식·게이트 기준·에이전트 역할 지침
```

- **하나의 작업 = 하나의 세션 계보.** 단계가 바뀌어도 세션은 이어진다(같은 담당이
  계속). **담당 에이전트가 바뀔 때만 인수인계** — 새 에이전트가 남겨진 기록을 읽고
  이어간다. (초안의 "단계마다 새 세션"은 사용자 교정으로 폐기.)
- **세션은 전체 역사가 아니라 제한 팩을 받는다**: 목표 + 현재 단계 지침 + 직전
  산출물 + 대화 요약. Planning Context Pack(하드 토큰 예산) 문법의 확장 — 토큰
  절약이 구조로 보장됨.
- **단계 지침은 열마다 하나, 모든 카드가 공유.** 예: 구현 열 = "patch + 기준 커밋 +
  SHA-256 + 검증 결과를 남긴다" (현행 Codex 전달 관행의 제품화). 기존 분류로 Agent
  Skill/Policy에 해당.
- **세션 실행 = Git 동기화 파이프라인의 일반화**: 카드 "시작" → 서명된 일회용 작업
  계획 → 로컬 커넥터 큐 → CLI 세션 시작(팩 주입) → 에이전트가 MCP로 카드에 결과
  기록. 전부 기존 부품의 재배선. 자동 세션 실행은 터미널 기반 커넥터로는 **내부
  도그푸딩 전용**, 상용은 서명된 데스크톱 헬퍼(출시 게이트 2) 이후.
- **대화↔세션 연결 = "문자로 기록하고 통화처럼 보여주기"**: 모든 대화·기록의
  원본은 DB(비동기 큐, 영속 — 근거·감사·재개의 기반)에 남기고, 세션이 살아 있는
  동안 그 위에 라이브 터미널 스트림을 표시 계층으로 얹는다. 연결이 끊겨도 기록은
  유실되지 않는다.

## 4B. 저장 (라운드 4 확정)

카드·작업·대화·게이트 기록은 **새 DB 테이블**(기존 Supabase·RLS·Realtime 재사용)에
저장한다. 캔버스 그림이 아니라 사업 기록이므로 캔버스 JSON·병합·undo와 분리한다.
캔버스 실체와는 ID 참조로 연결.

## 4C. 열별 통과 산출물 — 여권의 도장 (라운드 6 채택, 소프트 권고 형식)

게이트 승인은 "느낌"이 아니라 "산출물 확인"을 지향한다. AI 산출물은 declared로
시작하고 검증이 붙어야 한다(Reality Level 문법 재사용). 소프트 게이트이므로 아래는
권고 형식이며, 없이 통과하면 강행 기록이 남는다.

| 열 | 나갈 때의 도장 | 현행 도그푸딩 실체 |
|---|---|---|
| 백로그 → A | 목표 한 줄 + 이유 | "파운드리 모델 기획하자" |
| A 기획 → | 기획서/설계의뢰서 (목표·범위·성공 기준·구현 지시) | CODEX_BRIEF, 블루프린트 |
| B 개발 → | patch + 기준 커밋 + SHA-256 + 검증 결과 | Codex 전달 관행 그대로 |
| C 배포 → | 배포 기록 + 배포 후 실제 확인 (사용자 E2E) | Vercel 배포 + 화면 확인 |
| D 운영 | (통과가 아니라 상주 가능) 관측 연결·상태 기록 | LiveOps 관측 |
| E~H | 자유 형식 기록 (Connector 연동 전까지 수동) | — |

검수 열이 따로 없는 이유: A~H 8열 결정에서 **검수 = 게이트 그 자체**다. B를 나가는
승인 행위가 곧 검수다.

## 7. 첫 구현 슬라이스 — "기록 먼저" (0.5.0 구현 상태)

**1차 (MVP, 코드 구현·검증 완료):** 보드 + 백로그 + 8열 + 목표/작업 트리(접기·파생 가지) + 수동 소프트
게이트 + 종착 열 지정 + 산출물 첨부 + 대화 기록(세션 자동 실행 없이 수동 브리지 —
컨텍스트 팩을 복사해 CLI에 붙이고, 결과 보고를 카드에 기록). 먼저 현행 협업(사용자 +
Claude Code + Codex)을 보드에 **기록**하며 구조를 검증한다.

구현 근거는 `supabase-workshop.sql`, `shared/workshop.js`, `src/components/WorkshopBoard.jsx`,
`src/lib/workshopApi.js`, `mcp/workshopStore.js`와 Workshop 전용 회귀 테스트다. 활성화에는
기존 `supabase-schema.sql`, `supabase-shares.sql` 다음 `supabase-workshop.sql`을 사용자가
Supabase SQL Editor에서 실행하고, 검증된 소스 커밋을 배포하는 운영 단계가 남아 있다.

**2차(계속 미구현):** 세션 자동 실행(서명 계획→커넥터 큐→CLI, 내부 도그푸딩 전용) + 라이브
스트림 표시 계층 + 캔버스 실체 참조 연결.

**이후(계속 미구현):** 열별 자동 통과 설정, 대화 트리 fork 세션 재개, E~H Connector,
시야 제한 참여자 redaction. 1차 RLS는 캔버스 소유자와 수락 참여자에게 같은 보드
읽기·쓰기 범위를 주며, 이 단순화는 `TECHNICAL_DEBT.md`의 `WORK-002`에서 추적한다.

## 5. 기존 자산과의 정합 (조립표)

| 새 기능 | 재사용하는 기존 자산 |

| 자동 제안 카드 | LiveOps 관측, Reconciliation Finding |
| 예스코드 배정분 | Safe Operations (Operation Lifecycle) |
| A 기획 열의 산출물 | Starting Protocol — Project Master |
