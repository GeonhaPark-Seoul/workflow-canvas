# 트윈 정체성과 관측 상태

Workflow Canvas는 노드가 **무엇을 표현하는지**와 **어디까지 실제 근거에 연결됐는지**를 별도 축으로 관리한다. 제품 엔진처럼 코드 안의 논리 구성요소와 Vercel 배포처럼 실행 중인 자원을 같은 `LIVE/비-LIVE` 한 줄로 표시하면 사용자가 디지털 트윈 여부를 잘못 이해하기 때문이다.

## 축 1: 표현 대상

- `논리 구성`: Twin Core, Create Graph와 같은 제품 엔진, Contract, Builder, Pipeline 등이다. 실제 독립 서버라고 주장하지 않는다.
- `설계 실체`: 앱, API, DB, 배포, 사용자 역할처럼 캔버스에 모델링됐지만 아직 외부 실행 근거가 없는 대상이다.
- `실행 실체`: 서버가 허용된 커넥터로 방금 상태를 확인한 실제 자원이다.

## 축 2: 근거 연결 수준

- `연결 안 됨`: 수동 도형이거나 아직 TwinBuild 식별자와 연결되지 않은 오래된 지도 노드다.
- `CODE`: 코드, manifest 또는 Connector 발견 결과의 안정된 실체 ID와 지문에 연결된 스냅샷이다. 실행 상태 확인을 뜻하지 않는다.
- `LIVE`: 서버가 제공한 자원 ID, 관측 시각과 검증 결과가 있는 실행 관측이다. 브라우저가 보낸 값만으로는 승격할 수 없다.
- `오래됨/부분 확인/오류`: 이전 LIVE 관측의 신선도 또는 실행 검사 결과다.

따라서 제품 엔진은 `논리 구성 + CODE`, React Flow 캔버스 엔진은 `설계 실체 + CODE`, Vercel 배포는 평소 `CODE`이며 서버 확인 직후 `LIVE`가 될 수 있다.

## 바인딩 안전 계약

`bind_node` 수정안은 일반 노드 편집 권한이 아니다.

1. `digitalTwinBinding` 한 필드만 추가하거나 갱신한다.
2. 노드 ID, 종류, 상위 그룹, 공급자, 외부 참조와 기존 바인딩을 포함한 정체성 지문이 미리보기와 같아야 한다.
3. 다른 Source나 Entity에 이미 연결된 노드는 자동으로 바꾸지 않는다.
4. 위치, 크기, 제목, 설명, 메모, 사용자 파츠와 연결선은 보존한다.
5. 최대 24개를 한 검토안으로 묶으며, 적용 후 남은 대상은 다음 검토안으로 나눈다.
6. `digitalTwinBinding`은 코드 스냅샷 근거다. 서버 전용 `twinRuntime`을 대신하거나 LIVE를 위조할 수 없다.

## 현재 한계

- CODE 지문이 최신 코드와 다른지는 Twin Core 재검사 결과로 판정한다. 지속 실행 에이전트가 자동으로 갱신하지 않는다.
- 논리 엔진의 실제 함수 호출량, 성공률과 지연 시간은 아직 런타임 관측 대상이 아니다.
- 공개 신뢰 증명과 종단간 암호화는 별도 출시 부채이며 CODE/LIVE 표시가 이를 보증하지 않는다.
