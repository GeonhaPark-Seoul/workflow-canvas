## 1. 왜 파운드리를 참조 모델로 삼는가

Workflow Canvas OS의 중심 문장("모든 것은 Asset이 되고…")은 이미 온톨로지 선언이다.
twin 관련 문서에서 반복된 용어 혼란(entity/Asset, Twin Build, Adapter)은 **목표 참조
모델이 없는 상태에서 단계마다 이름을 새로 지어온 결과**였다. 파운드리는 같은 문제
(흩어진 시스템을 하나의 조작 가능한 모형으로 만드는 것)를 이미 상용 규모로 풀어낸
유일한 사례이므로, 이름과 구조를 빌려 정렬하면 앞으로의 용어 발명 비용을 줄일 수 있다.

**주의:** 파운드리는 대기업 전용 상품이고 우리는 아니다. 파운드리의 인프라 규모·
거버넌스 무게를 따라갈 필요는 없다. 빌리는 것은 **개념 구조**이지 **구현 규모**가
아니다.

## 2. 파운드리 6층 구조와 대응

파운드리가 파는 것은 "조직의 디지털 모형"이며, 6개 층으로 쌓는다.

| 층 | 하는 일 | 파운드리 용어 | Workflow Canvas 대응 | 상태 |
|---|---|---|---|---|
| ① 연결 | 원본 시스템에서 근거를 가져온다 | Data Connection | Connector Bridge / Connector | 알파 |
| ② 변환 | 씻고 합치고, 매 단계를 기록한다 | Pipeline / Transform / Build, Lineage | Asset Build, Evidence | 부분 — 계보 화면 없음 |
| ③ 온톨로지 | 데이터를 현실의 대상·관계·행동으로 번역한다 | Ontology (Object Type/Object/Link/Action) | Asset 원장(Asset Base) — Asset/Relation/Operation | 부분 — Asset 종류(facet)가 관리 화면 없이 계획만 |
| ④ 앱 | 사람이 온톨로지를 보고 조작하는 화면 | Workshop/Quiver/Object Explorer | 캔버스 + Lens + 층(Layer)·오버레이 | 있음 — **차별점** |
| ⑤ 액션 | 승인·권한 통제를 거쳐 원본 시스템에 되쓴다 | Action Type | Operation / Safe Operations | 있음, Engine 0.x라 범위 좁음 |
| ⑥ AI | 사람과 같은 온톨로지·같은 액션만 쓰는 AI | AIP | AI 오케스트레이션(§9 미래) + MCP Gateway | 계획 |

파운드리의 해자는 ③ 단독이 아니라 **③(모형)과 ⑤(되쓰기)가 닫힌 루프**를 이루는 것이다.
보기만 하는 모형은 BI 대시보드와 다르지 않다.

## 3. 온톨로지란 무엇인가

온톨로지(ontology)는 "세상에 무엇이 존재하는가에 대한 목록과 문법"이다. 데이터
세계에서는 **다루는 대상의 종류, 각 종류의 속성, 종류 사이의 관계, 각 종류에 할 수
있는 행동을 정의한 사전**으로 번역된다.

핵심 구분은 **종류(type)와 실체(instance)**다. "고객"이라는 종류를 한 번 정의하면
"김철수", "이영희"는 그 종류의 실체다. 파운드리는 종류를 Object Type, 실체를 Object로
부른다.

MASTER §1의 중심 문장과 §2.1 Asset 문법 표는 이미 온톨로지 선언이자 정의다.

| 파운드리 | Workflow Canvas |
|---|---|
| Object (대상 실체) | **Asset** |
| Object Type (대상 종류) | **Asset 종류** — 조합형(facet), 관리 화면 미구현 |
| Link (관계) | **Relation** |
| Action (통제된 행동) | **Operation** |

### 온톨로지가 어려운 진짜 이유

개념은 쉽고 운영이 어렵다. 어려운 지점은 네 가지다.

1. **동일성** — 서로 다른 시스템의 서로 다른 이름이 같은 대상임을 정확히 판정하는 것
2. **드리프트** — 현실은 계속 바뀌는데 모형이 낡지 않았음을 어떻게 보증하는가
3. **되쓰기 안전** — 모형에서 누른 버튼이 실제 시스템을 망가뜨리지 않게 하는 통제
4. **합의** — "고객이란 무엇인가"에 조직 전체가 동의하게 만드는 것 (파운드리는 이걸
   현장 파견 인력으로 푼다 — 소프트웨어가 아니라 노동으로)

Workflow Canvas는 1~3을 이미 원칙 수준에서 풀어놨다: `same_as`는 근거+사람 승인
필수(불변 원칙 7), Reality Level·stale이 드리프트를 관리(불변 원칙 1), Operation
Lifecycle이 되쓰기를 통제(불변 원칙 2). **이 세 원칙의 엄격함 — 파운드리조차 사용자
1급 개념으로 노출하지 않는 "주장/사실 분리 축" — 이 해자 후보 1번이다.**

## 4. 이름 3층 — wire 문제는 온톨로지 문제다

"같은 대상이 곳곳에서 다른 이름을 갖는다"는 온톨로지가 푸는 문제이고, wire 문제는
그 축소판이다. 파운드리는 모든 대상에 이름을 3층으로 준다.

| 층 | 성질 | 파운드리 | Workflow Canvas 대응 |
|---|---|---|---|
| 내부 ID | 절대 불변, 기계용 | RID | Asset 안정 식별자 (기존) |
| API 이름 | 코드가 쓰는 이름, 거의 안 바꿈 | API name | **wire 이름** |
| 표시 이름 | 사람이 보는 이름, 언제든 변경 | Display name | 표시 이름 |

표시 이름은 마음대로 바꿔도 아무것도 깨지지 않는다 — 코드와 저장은 아래 두 층만
보기 때문이다. 원본 시스템의 이름들(예: `cust_id`)은 온톨로지 관리 화면에서
"어느 원본 열이 어느 속성에 연결되는가" 매핑으로 보여준다.

**적용:** 원장(Asset Base)에 저장되는 모든 Asset 레코드가 세 이름을 필드로 가진다.

```
{
  id: "wc-asset-01H8...",        // 내부 ID (불변)
  wire: "entity",                // wire 이름 (스키마 버전에서만 변경)
  displayName: "로그인 기능",     // 표시 이름 (언제든 변경)
  sourceNames: ["auth.signIn", "login_handler"]  // 원본 시스템 이름들
}
```

파운드리는 온톨로지 정의 자체도 문서가 아니라 플랫폼 안의 데이터로 저장한다.
같은 원칙으로 **Asset 종류 정의와 용어 사전(현재 MASTER §2.5 매핑표의 후신)도
원장의 데이터로 이관**하는 것이 F1(§9)의 목표다. 이렇게 되면 "단어를 클릭하면
[표시 이름/wire 이름/과거 이름/원본 이름] 카드가 뜬다"는 UI가, 이미 원장에 있는
데이터를 보여주는 것만으로 자연히 성립한다.

## 5. "선언 vs 검증" 문제 — 파운드리와 다르게 간다

파운드리는 이 축을 우리만큼 엄격하게 나누지 않는다. 처리 방식은 두 겹이다.

1. **계보(lineage)** — 모든 Object는 어느 데이터셋에서, 어느 빌드가, 언제, 어느
   원본에서 만들었는지 항상 추적 가능하다. "왜 믿는가"의 답은 항상 "출처를
   따라가 보라"다.
2. **신선도·건강 감시** — 데이터셋마다 "마지막 성공 빌드 시각"과 검사 규칙
   (예: 6시간 내 갱신, 빈 값 금지)이 붙어 위반 시 경보가 울린다.

Object 하나하나에 "이건 주장이고 저건 검증됐다"는 등급은 없다. 파운드리 고객의
데이터가 기간계 시스템(ERP, 센서)에서 파이프라인으로 들어오므로, 근거 없이 사람이
선언만 한 Object가 애초에 거의 없기 때문이다.

**우리는 다르게 가야 한다.** 우리 고객(바이브 코더)의 "시스템 구조도"는 AI가
그려준 것이라 **선언과 실제가 어긋나는 게 기본값**이다. `declared → discovered →
observed → runtime-verified`(Reality Level)는 파운드리에 없는, 우리 시장에서만
필요한 엄격함이다. **여기는 파운드리를 따라가면 안 되는 지점**이며, 우리 해자
후보 1번의 실체이기도 하다.

파운드리 작동 원리의 이름 어디에도 "twin"이 들어가지 않는다는 점도 확인됐다.
과정은 build/sync, 결과물은 ontology/dataset/object로 부르고, "digital twin"은
제품 전체를 소개하는 마케팅 문장에서만 느슨하게 쓴다. §7의 슬롯 5 결정(내부
메커니즘에서 twin 폐기, 대외 소개 문장에만 생존)은 이 선례를 그대로 따른다.

## 6. 우리만의 확장 — 노드 → 그룹 → 층 → 건물 → 마을

파운드리 ④ 앱층은 결국 폼·표·대시보드다. 공간이 없다. Workflow Canvas의
노드→그룹→층(Layer) 문법을 그 위로 두 계단 더 얹는 것이 파운드리가 시도하지 않은
방향이며, 해자 후보 2번이다.

| 계단 | 대응 |
|---|---|
| 노드 | Asset 하나 (기존) |
| 그룹 | 관련 Asset 묶음 (기존) |
| 층(Layer) | 추상화 깊이 축, L1~L4 (기존, §2.4) |
| **건물** | 시스템 지도 하나 전체 (소프트웨어 한 개) — 신규 |
| **마을** | 여러 시스템 지도의 포트폴리오, 건물 간 연결선 = Gateway — 신규 |

기존 결정(3D는 표시 방식이지 데이터 모델이 아니다, Z축 의미는 하나)은 유지한다.
**건물·마을은 먼저 계층 데이터로 정의**하고, 3D는 그 데이터를 그리는 렌더러로
나중에 붙인다. 그러면 3D 구현이 늦어져도 데이터 모델은 앞서갈 수 있다.

UI 세부(캔버스에서 실제로 어떻게 보이고 조작되는가)는 §9의 F0~F2 완료 후 별도
논의로 다룬다.

## 7. 최종 용어 확정표

| 슬롯 | 이전 표현 | 확정 용어 | 비고 |
|---|---|---|---|
| A | 산문 속 `entity` | **Asset** | 세분화는 새 단어가 아니라 Asset 종류(facet)로 |
| 1 | `TwinBuild` + "Asset Graph"(개념) | **Asset 원장 (Asset Base)** | 정규형·개념 통합 |
| 2 | Twin Build(과정) | **Asset Build** | 근거→Asset 원장 변환 |
| 3 | Twin Core(엔진) | **Asset Core** | 정규화·동일성·대조 |
| 4 | Create Graph(엔진) | **Draw Map** | 순수 가시화 전용 — 배치·연결선·그룹만 담당, 온톨로지·사실 판정과 무관 |
| 5 | Digital Twin/트윈(내부 메커니즘) | **내부 폐기** | "디지털 트윈"은 대외 소개 문장 전용. 검증 상태는 기존 **LIVE**(Reality Level `runtime-verified`)로 표현 |
| 6 | Graph(범용) | **지도** | Graph Layout→지도 배치, Graph Analysis Adapter→지도 분석 Adapter, graph health→지도 상태. 기존 "시스템 지도"와 같은 계열이라 충돌 없이 강화됨 |
| 7 | wire | **유지** | "OOO의 wire 이름은 XXX다" |
| 8 | (레거시 Twin Adapter Contract) | **새 이름 없음** | 분해(Connector + Stage별 Adapter) 전까지 현재 코드명 유지 |
| D | Adapter/Connector/Lens/Proposal/Operation/Graphify | 변경 없음 | |

## 8. 갭 분석

| 갭 | 현재 상태 | 필요한 것 |
|---|---|---|
| Asset 종류 관리 | 조합형으로 계획만, 관리 화면 없음 | 종류를 보고·정의·편집하는 화면 (F1) |
| 계보 가시화 | Evidence는 존재하나 사용자 화면 없음 | Asset→근거 사슬을 따라가는 오버레이 (F2) |
| Operation 범위 | Engine 0.x, 읽기·대조만 | 안전한 핵심 조작부터 단계적 확대 (F3, 기존 로드맵과 동일) |

| 대규모 객체 탐색·시계열 | 없음 | 장기 갭, 현재 시장에 불필요 |

## 9. F-로드맵 제안

우선순위가 아니라 **의존 순서**다. 뒤 단계가 앞 단계의 산출물을 딛고 선다.

- **F0 용어 정렬** — 이 문서 + MASTER 반영. 이후 모든 문서·코드·설계의뢰서가 이
  단어를 쓰므로 가장 싸고 가장 먼저.
- **F1 Asset 종류 관리자** — §3·§4의 첫 번째 차이(종류가 1급 시민이 아님)를 메운다.
  F3(조작)이 종류에 붙어야 하고, 동일성 판정·검색도 종류가 있어야 하며, 스키마
  v4 개편(entity→asset wire 개명)과 한 배에 태울 수 있다.
- **F2 근거·계보 가시화** — F1 다음인 이유: 계보는 "어떤 종류의 근거가 어떤 종류의
  Asset을 만들었나"라 종류 체계가 먼저 있으면 단순해진다. 불변 원칙 1(주장과
  사실의 분리)을 사용자 눈에 보이게 만드는 일이라 제품 차별점의 완성이기도 하다.
- **F3 조작 확대** — 기존 로드맵의 Engine 1.x와 동일 항목. F1의 종류에 조작을
  붙인다.
- **F4 소프트웨어 캔버스 구동** — 원래 목표. 뒤에 있는 이유는 미루는 게 아니라
  F1~F3이 부품이기 때문이다: (a) 시스템을 올리고(온보딩 프로토콜), (b) 상태를
  관측하고(LiveOps), (c) 캔버스에서 실제 행동을 실행(F3 조작)해야 "구동"이 된다.
  부품 없이 먼저 하면 twin 문서들이 겪은 임시 용어·임시 구조가 재발한다.
  - F4 후보 기능 — **라이브 프리뷰**: 사용자가 캔버스에서 등록 UI 상수를 고치면
    타겟 앱의 실제 화면이 실시간으로 바뀌는 것. 신규 의존성 없이 로컬 Vite
    HMR(사용자 앱의 dev 서버) + 기존 콘텐츠 노드(브라우저) + SL-3 왕복 편집을
    조립한다. 미리보기는 노코드/완전 자동, 커밋·배포는 기존 2중 승인 게이트
    유지(§10 축 분리). 도입 시기는 기획에서 결정.
- **F5 AI** — 마지막인 이유는 불변 원칙 8(결정적 자동화 우선)이 이미 답했다. AI가
  안전하려면 조작 계약·승인·감사가 먼저 있어야 하고, 파운드리도 AIP를 같은
  순서로 맨 마지막에 얹었다.

이 문서의 F-로드맵은 MASTER §9(로드맵)와 별개 트랙이 아니라, 확정되면 §9에
병합·재배열되어야 한다. 현재 §9의 Source Lens 프로그램(SL-5 등)과의 우선순위
조정은 별도 논의가 필요하다.

## 10. Stage 분류 문법 — 실행 방식 × 승인 루트

제품의 모든 Stage와 Operation은 아래 2축 라벨을 갖는다. 새 개념이 아니라 MASTER에
흩어져 있던 축(불변 원칙 8, Operation Lifecycle, AI 오케스트레이션 실행 방식)을
라벨로 압축한 것이다.

**축 1 — 실행 방식: 그 일을 실제로 누가·무엇이 하는가**

| 값 | 뜻 |
|---|---|
| **수동** | 사람이 직접 한다 |
| **예스코드** | 결정적 코드 자동화가 한다 (규칙·AST·서버 — AI 아님) |
| **노코드** | AI가 사람 대신 한다 |

**축 2 — 승인 루트: 자동화(예스코드·노코드)가 진행될 때 사람 동의가 필요한가**

| 값 | 뜻 |
|---|---|
| **완전 자동** | 동의된 범위 안에서 사람 개입 없이 진행 |
| **웹 승인** | 진행 전 화면에서 1차 승인 |
| **2중 승인** | 웹 + 터미널(미래: 서명된 헬퍼 다이얼로그)의 이중 게이트 |

같은 자동화라도 위험도에 따라 승인 루트가 다르다. 승인 루트의 "터미널"을 서명된
데스크톱 헬퍼로 바꾸는 것은 접근성 개선이지 승인 제거가 아니다.

### 현재 제품 분류 (2026-07-20 기준)

시스템 온보딩: 연결·동의(수동/—) → 근거 수집(예스코드/완전 자동, 동의 범위 안) →
정적 분석(예스코드/완전 자동) → 기능군 정합화 SL-5 계획(**노코드** 후보 생성 +
예스코드 검증/검증이 게이트) → 동일성(예스코드/**same_as만 사람 승인**) →
신뢰경계(예스코드+선언은 수동/웹 승인) → 검토 Proposal 생성(예스코드/**웹 승인
필수**) → 실체화(예스코드/승인 후 완전 자동) → 첫 관측(예스코드, 서버만 LIVE
승격/완전 자동).

운영 조작: 캔버스 편집(수동), AI 캔버스 생성(**노코드**/사용자 수용), Git
동기화(예스코드/2중 승인), 소스 수정·롤백(예스코드/2중 승인), SQL 실행(**수동** —
사용자가 직접), AI 코드 설명 파일럿(**노코드**/기본 비활성).

### 세 가지 함의

1. **현재 제품은 예스코드가 지배적이고, 노코드(AI 대행)는 3곳뿐이다** (AI 캔버스
   생성, 설명 파일럿, SL-5 후보). 이게 설계 의도다(불변 원칙 8: 결정적 자동화 우선,
   AI는 해석·계획·예외에만).
2. **로드맵의 실질 의미 하나는 수동 칸을 자동화 칸으로 옮기는 것이다.** SQL 직접
   실행→승인된 마이그레이션 조작(Engine 3.x), 연결·동의의 터미널 설치→서명된
   데스크톱 헬퍼(출시 게이트 2). 이 표는 분류표이자 출시 준비도 지도다.
3. **노코드(AI)의 확대는 승인 루트 축을 타고 온다.** AI 오케스트레이션 연결
   순서(일회성 읽기→수정안→승인된 조작→예약형→상시형)는 결국 "노코드 칸을
   넓히되 승인 루트를 단계적으로만 완화한다"는 뜻이다.

### 유지 규칙 (제안 — MASTER 반영 대상)

- 모든 새 Stage·Operation 정의는 2축 라벨(실행 방식 / 승인 루트)을 필수 필드로
  갖는다 (프로토콜 문서 Stage 표와 Operation manifest에 열 추가).
- 온보딩 Workflow Run 진행 화면의 각 Work Item에 라벨 배지를 노출한다 (UI 논의
  의제).

## 11. 다음 논의 — UI (미착수)

아래는 시드일 뿐이며 다음 세션에서 사용자와 구체화한다.

1. **종류 관리자(F1) = "종류 지도" 캔버스** — 별도 관리자 앱이 아니라 종류 자체를
   노드로 그린 캔버스. 종류가 노드, 조합(facet)·상속이 연결선. 개별 Asset 노드의
   종류 배지를 클릭하면 그 종류 노드로 점프.
2. **계보(F2) = 오버레이** — 기존 오버레이 문법(§2.4)에 "계보 오버레이" 추가.
   Asset 선택 시 근거 사슬(관측→커넥터→원본)을 연결선 하이라이트로 표시하고,
   §4의 이름 카드(표시/wire/과거/원본 이름)를 함께 노출.
3. **건물·마을 = LOD의 위 두 계단** — 시스템 지도 전체를 축소하면 건물 한 채로
   접히고(건물=시스템 지도 1개), 마을 뷰의 건물 간 연결선은 시스템 간 Gateway.
   3D 렌더는 나중, 데이터 계층(§6)은 F1 종류 체계에 먼저 정의.
